# Инструкции для Codex

Эти правила применяются ко всем задачам в репозитории. Пользователю достаточно попросить: «прочти `AGENTS.md`» — после этого Codex должен самостоятельно выполнить стартовый протокол ниже.

## Стартовый протокол

До анализа и изменений:

1. Полностью прочитать `docs/PROJECT_CONTEXT.md` — это каноническая сводка текущего состояния, архитектуры, истории, ограничений и команд проверки.
2. Прочитать свежий раздел `CHANGELOG.md`, затем выполнить `git status --short --branch` и `git log --oneline -15`, чтобы учесть изменения после последней сводки.
3. Если задача затрагивает конкретную подсистему, открыть указанные для неё документы из таблицы в `docs/PROJECT_CONTEXT.md`. Старые планы и решения не считать текущей реализацией без проверки кода.
4. Для поиска и анализа кода сначала использовать граф `codebase-memory-mcp`: `search_graph`, `trace_path`, `get_code_snippet`, затем при необходимости `query_graph` и `get_architecture`. Для строк, конфигурации, документации и при недостаточных данных графа использовать обычный поиск.
5. Если граф отсутствует или не соответствует текущему `HEAD`, индексировать репозиторий через `codebase-memory-mcp`, проверить результат и получить краткий обзор через `get_architecture`.

## Обязательное ведение истории

- После каждого значимого изменения кода, конфигурации, инфраструктуры или поведения добавить в верхний датированный раздел `CHANGELOG.md` краткую запись: что изменено и зачем. Не ждать напоминания пользователя.
- Если изменились архитектура, актуальные возможности, ключевые ограничения, production-состояние, важные решения, команды запуска/проверки или следующий приоритет, одновременно обновить `docs/PROJECT_CONTEXT.md`.
- Не превращать `PROJECT_CONTEXT.md` в покоммитный журнал: подробная хронология остаётся в `CHANGELOG.md` и Git, а контекст хранит только актуальную сводку и важные вехи.
- Перед завершением задачи проверить, не устарели ли затронутые `README.md`, документы в `docs/`, env-примеры и deploy-runbook. Обновлять только связанные с задачей материалы.
- В финальном ответе перечислить выполненные проверки, коммит и результат push. Нельзя объявлять значимую задачу завершённой, забыв обновить историю.

## Git и безопасность

- После важных изменений сделать отдельный осмысленный commit и `git push`, если пользователь явно не запретил Git-операции.
- Перед коммитом проверить `git status` и diff; включать только относящиеся к задаче изменения. Существующие пользовательские изменения сохранять и не присваивать.
- Если push невозможен из-за remote, расхождения веток или прав, не переписывать историю без разрешения и явно сообщить причину.
- Никогда не коммитить `.env`, production-секреты, токены, сертификаты с приватными ключами, дампы и локальные backup-файлы.
- Папка `kornix base/` содержит копии внешних репозиториев `meteo` и `kornix_site`: только читать, никогда не изменять и не добавлять в Git.
- Не выполнять production deploy, изменение Caddy, MAX-настроек или внешних сервисов без явного запроса пользователя.

## Технические границы

- Бот и Mini App работают с KORNIX только через backend HTTP API; прямой доступ к БД и перенос расчётной логики KORNIX в этот репозиторий запрещены.
- Все KORNIX-запросы должны идти через `src/kornix/kornixClient.ts`; запись поливов и осадков — через общий `src/kornix/operationService.ts`.
- Проверку MAX webhook/initData, session security, managed scope, защиту секретов в логах и production fail-closed проверки нельзя ослаблять ради упрощения.
- Каталоги `dist/`, `dist-test/`, `dist-miniapp/`, `node_modules/` — генерируемые результаты, не источники истины.
